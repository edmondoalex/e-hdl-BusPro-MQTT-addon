1) Mappa mentale del sistema (molto importante)

Pensa all'addon come a 4 blocchi:

  1. BusPro UDP: parla con i moduli HDL (luci, cover, sensori).
  2. Store interno: salva configurazione (dispositivi, scenari, gruppi, trigger).
  3. MQTT Discovery: crea/aggiorna entita in Home Assistant.
  4. UI Web: Admin per configurare, pagine utente per usare.
Se capisci questo, capisci il 90% dell'addon.
Figura 1: schema a blocchi (A-B-C-D) usato in tutta la guida.
2) Primo avvio senza errori
  1. In opzioni addon imposta gateway_host e gateway_port.
  2. Controlla MQTT host/port corretti.
  3. Avvia addon e verifica pillole stato in header Admin (MQTT, Gateway, UDP).
  4. Se vedi entita vecchie/sbagliate in HA: usa Reset discovery con cautela.

Attenzione: Reset discovery rimuove retained config BusPro e poi ripubblica.

3) Dispositivi: come aggiungerli bene

Nel blocco Dispositivi inserisci: nome, gruppo UI, subnet/device/channel, categoria, icona, dimmable.

CampoUso praticoErrore comune
Gruppo (User UI)Ordine visivo pagine utenteLasciarlo vuoto e poi non trovare il device
CategoriaCome viene pubblicato in HA (light/switch ecc.)Mettere Switch e aspettarsi slider %
DimmableAbilita brightness (0..255)Mettere No su canale che vuoi regolare

Regola %: se vuoi percentuale in HA, il canale deve uscire come light, non switch.

Esempio 1 - Creo una VMC con slider %
  1. Nome: VMC FAN
  2. Addr: 1.185.60
  3. Categoria: custom Extra (non Switch)
  4. Dimmable: Si
  5. Risultato in HA: entita light.buspro_extra_vmc_fan con brightness
4) Scenari: RUN, ON/OFF, trigger (spiegato semplice)

RUN = esegui lo scenario come e stato salvato.
ON/OFF = usa la logica switch scenario (puo invertire in base alla configurazione scenario).

  • Per scenari affidabili (es. Vado Via, Sveglia) usa: RUN attivo, ON/OFF disattivo.
  • Per uso manuale tipo "toggle rapido", puoi lasciare ON/OFF attivo.

Trigger scenario:

  1. In Admin crea Trigger E-manager (nome friendly).
  2. Nello scenario (editor in pagina Luci) abilita Automazione E-manager e scegli trigger.
  3. In HA/Alexa chiama il trigger con button.press o publish MQTT.
Esempio 2 - Scenario "Vado via"
  • Luci: tutte OFF
  • Tapparelle: tutte Giu (close)
  • Flag consigliati: RUN=on, ON/OFF=off
  • Trigger E-manager: VADO_VIA_HA attivo
MQTT trigger: buspro/cmd/scenario_ha_trigger/VADO_VIA_HA payload RUN
5) Tapparelle: Su/Giu/Stop, Min, Step
  • Su = OPEN
  • Giu = CLOSE
  • Stop = STOP

Min/Step sono per rampa a impulsi, non per comando diretto classico:

  • Min = 0: comando diretto continuo (step ignorato).
  • Min alto + Step: alterna impulsi e stop, utile per movimento piu dolce/lungo.

Per "Vado via": di norma tutte su Giu e RUN.

6) Entita Home Assistant esterne

Sezione Home Assistant devices (UI utente) serve per mostrare entita HA in pagine utente addon.

  • Non crea un device BusPro nuovo.
  • Non passa da MQTT discovery BusPro: usa API HA.

Esempio: aggiungi light.sala in pagina extra per controllarla dalla UI addon.

7) MQTT discovery e pulizia
  • Ripubblica discovery: refresh normale, sicuro.
  • MQTT cleanup: pulizia mirata di config/state vecchi su un addr.
  • Reset discovery: reset forte retained + ripubblicazione completa.

Quando usarli:

  1. Rinomina leggera -> Ripubblica discovery.
  2. Cambio subnet/device/channel -> MQTT cleanup mirato.
  3. Entita fantasma persistenti -> Reset discovery.

9) Checklist finale "fatto bene"
  1. Gateway e MQTT online
  2. Dispositivi con addr corretti e categoria coerente
  3. Scenari testati con RUN
  4. Trigger E-manager collegati ai soli scenari necessari
  5. Backup esportato dopo le modifiche
